## Requirements
### Requirement: OpenSpec availability is detected per selected workspace
The system SHALL expose the mobile Open Spec entry only for the currently selected desktop-advertised workspace when the desktop agent finds an OpenSpec directory at the workspace root named `.OpenSpec`, `openspec`, `.openspec`, or `OpenSpec`.

#### Scenario: Workspace has a compatible OpenSpec directory
- **WHEN** the mobile user selects a workspace whose desktop agent finds one of the supported OpenSpec directory names at the workspace root
- **THEN** the mobile UI shows an enabled Open Spec entry for that workspace

#### Scenario: Workspace has no OpenSpec directory
- **WHEN** the mobile user selects a workspace without any supported OpenSpec directory at the workspace root
- **THEN** the mobile UI hides the Open Spec entry or presents it as unavailable without opening a progress panel

### Requirement: OpenSpec summaries are generated by the desktop agent inside workspace boundaries
The system SHALL generate OpenSpec summaries on the desktop agent from the selected allowlisted workspace and MUST NOT allow the mobile client to provide arbitrary filesystem paths for OpenSpec reading.

#### Scenario: Mobile requests OpenSpec summary
- **WHEN** the mobile client requests the OpenSpec summary for a selected `projectId`
- **THEN** the relay queues the request for the owning desktop agent and the desktop agent resolves only the fixed OpenSpec directory candidates inside the advertised workspace

#### Scenario: OpenSpec path would escape the workspace
- **WHEN** a supported OpenSpec directory or artifact path resolves outside the advertised workspace through traversal or symlink behavior
- **THEN** the desktop agent rejects or omits that path and the mobile response does not include escaped file data

### Requirement: OpenSpec progress is calculated from tasks files
The system SHALL calculate change progress from Markdown checkbox items in each change's `tasks.md` file.

#### Scenario: Change tasks include checked and unchecked items
- **WHEN** a change has a `tasks.md` containing `- [x]`, `- [X]`, and `- [ ]` checkbox items
- **THEN** the summary reports completed task count, total task count, percent complete, and read-only grouped tasks for that change

#### Scenario: Change has no task checkboxes
- **WHEN** a change has no `tasks.md` file or no Markdown checkbox task items
- **THEN** the summary keeps the change visible with a no-tasks status and excludes it from aggregate task completion math

### Requirement: Mobile users can review OpenSpec progress in a story timeline
The mobile UI SHALL present OpenSpec summary data as a mobile-friendly progress surface with overview metrics and a change timeline.

#### Scenario: User opens Open Spec panel
- **WHEN** the mobile user taps the Open Spec entry for an available workspace
- **THEN** the app opens a panel showing workspace context, OpenSpec directory name, aggregate task progress, change count, spec count, and a refresh action

#### Scenario: User reviews a change
- **WHEN** the summary includes one or more changes
- **THEN** each timeline row shows the change title or id, progress, task counts, concise proposal context when available, affected specs when known, and an expandable read-only task checklist

### Requirement: OpenSpec UI handles loading, empty, error, and offline states
The mobile Open Spec UI SHALL handle unavailable data without blocking the existing Codex workbench.

#### Scenario: Desktop agent is unavailable after a summary was cached
- **WHEN** the selected workspace has a cached OpenSpec summary and the desktop agent is offline or reconnecting
- **THEN** the Open Spec panel can show the cached summary as stale and provide a retry or refresh affordance

#### Scenario: Summary request fails without cache
- **WHEN** the mobile app cannot load an OpenSpec summary and has no cached summary for the selected workspace
- **THEN** the panel shows a concise empty or error state and the rest of the Codex workbench remains usable
